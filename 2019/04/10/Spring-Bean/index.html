Bean的常用配置

id和name

  • 一般情况下，装配一个Bean时，通过指定一个id属性作为Bean的名称
  • id属性在IOC容器中是唯一的
  • name其实也要是唯一的，同id的区别是可以存在特殊字符

class

  • 设置一个类的全路径的名称，主要作用是IOC容器生成类的实例

scope

类别 说明
singleton(默认) 在SpringIOC容器中仅存在一个Bean实例，Bean以单实例的方式存在
prototype 每次调用getBean()时都会返回一个新的实例
request 每个HTTP请求都会创建一个新的Bean，该作用域仅适用于WebApplicationContext环境
session 同一个HTTP session共享一个Bean，不同的HTTP session使用不同的Bean。该作用域仅适用于WebApplicationContext环境

这里需要注意的是一般情况下使用singleton。但是当Spring整合Struts时候，由于Action是多例的，这时就需要使用Prototype了

Bean的定义和初始化

Bean的定义和初始化在Spring IOC容器中是两大步骤，他是先定义，然后初始化和依赖注入。

Bean的定义分为三步：

  1. Resourse定位：这步是Spring IOC容器根据开发者的配置，进行资源定位，在Spring的开发中。通过XML或者注解都是十分常见的方式，定位的内容是由开发者提供的。
  2. BeanDefinition的载入，这个时候只是将Resourse定位到的信息，保存到Bean定义中（BeanDefinition）中，此时并不会创建Bean的示例。
  3. BeanDefinition的注册：这个过程就是将BeanDefinition的信息发布到Spring IOC容器中，注意，此时依旧没有对应的Bean的示例创建。

做完了这三步，Bean就在Spring容器中被定义了，而没有被初始化，更没有完成依赖注入，也就是没有注入其配置的资源给Bean，那么Bean它还不能完全使用。对于初始化和依赖注入，Spring Bean还有一个配置选项———-

lazy-init，其含义就是是否延时初始化Spring Bean。在没有任何配置的情况下，它的默认值为dafault，实际值为false，也就是Spring IOC默认会自动初始化Bean。如果将其设置为true，那么只有当我们使用Spring IOC容器的getBean方法获取它时，他才会进行Bean的初始化，完成依赖注入。

Bean的生命周期

Spring IOC容器的本质目的就是为了管理Bean，对于Bean而言，在容器中存在其生命周期，他的初始化和销毁也需要一个过程，在一些需要自定义的过程中，我们可以插入代码去改变他们的一些行为，以满足特别的需求，这就需要使用Spring Bean生命周期的知识了。下图展示了Spring IOC容器初始化和销毁Bean的过程。

由上图可以看到，Spring IOC容器对Bean的管理还是比较复杂的，Spring IOC容器在执行了初始化和依赖注入后，会执行一定的步骤来完成初始化，通过这些步骤我们就能自定义初始化，而在Spring IOC容器正常关闭的时候，他也会执行一定的步骤来关闭容器，释放资源。

下面对这些步骤进行详细的介绍。

  1. instantiate bean对象实例化

    即调用类的构造方法。

  2. populate properties封装属性

    即调用类的属性设置方法。

  3. 如果Bean实现BeanNameAware，则执行setBeanName

    即为该bean设置name的时候调用，可以想象为容器为map，name就是那个key。

  4. 如果Bean实现BeanFactoryAware或者ApplicationContextAware设置工厂，则执行setBeanFactory或者上下文对象setApplicationContext

    即为该bean设置容器的时候调用。

  5. 如果存在类实现BeanPostProcessor（前处理Bean），执行postProcessBeforeInitialization

    即在调用init-method之前调用。

  6. 如果bean实现InitializingBean，则实行afterPropertiesSet

    属性设置后调用。

  7. 调用init-method="setup"指定的初始化方法

    执行init方法。

  8. 如果存在类实现BeanPostProcessor（后处理Bean），执行postProcessAfterInitialization

    即在调用init-method之后调用。

  9. 执行业务处理

    即调用bean中的业务方法。

  10. 如果Bean实现DisposableBean，执行destory方法

    spring自身中的销毁方法。

  11. 调用destroy-method="teardown"指定的销毁方法

    执行destory方法。

需要注意的是：有些步骤是需要在一些条件下才会执行的。还有就是有些接口只是针对单个Bean而言的；BeanPostProcessor接口则是针对所有的Bean而言的。当一个Bean实现了BeanPostProcessor接口，我们只需要在Spring IOC容器中定义它就可以了，Spring IOC容器会自动识别并调用。
